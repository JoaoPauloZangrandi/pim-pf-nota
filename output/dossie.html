Dossiê do projeto — Esteira PIM-PF

Tutorial 5 · “Saiu a Pesquisa Industrial Mensal” · documento autossuficiente que explica todo o projeto. Mês de referência do exemplo: abril 2026.
Resumo executivo

Uma esteira 100% automatizada que, a partir da divulgação da PIM-PF no SIDRA/IBGE, coleta os dados, trata, analisa e gera uma nota informativa em PDF (via LaTeX) e relatórios HTML — em segundos, por um único comando. O critério central é a robustez: a esteira nunca falha em silêncio nem entrega um PDF errado; ou produz a nota correta, ou falha com mensagem acionável e log.

Python 3.11+SIDRA/IBGE pandas + panderaJinja2 → LaTeX → PDF tenacity (retry)pytest + CI Martin · Código Limpo, Cap. 7

1 · O desafio

Uma consultoria de macroeconomia precisa divulgar, até as 10h, uma nota sobre a PIM-PF que o IBGE publica por volta das 9h no SIDRA. O prazo é apertado, então o processo — coleta, tratamento, análise e geração do documento — precisa estar automatizado. O documento final é um PDF (a consultoria produz seus relatórios internamente em LaTeX) e deve ser visualmente limpo, informativo e sucinto. Acima de tudo: minimizar o risco de, no dia da divulgação, o código não funcionar.

4 · Arquitetura e módulos

Separação de preocupações: cada módulo tem uma responsabilidade única. A dependência da API de terceiros fica isolada num só lugar (o wrapper), e os códigos de tabela ficam todos em config.py (trocar um código = uma linha).

MóduloResponsabilidade
config.pyCentraliza códigos de tabela/variáveis/classificações do SIDRA, caminhos e parâmetros de rede. Troca de código = uma linha.
exceptions.pyHierarquia de exceções de domínio (raiz PimReportError) + guard exigir_nao_nulo. Princípios 1, 3, 4, 5, 8.
special_cases.pyPeriodoNaoPublicado (Special Case de Fowler). Princípios 6 e 7.
sidra_client.pyWrapper da API: timeout, retry exponencial (tenacity) só em transitórios, e tradução de erros para SidraError. Princípio 5.

pipeline.pyOrquestra tudo com try/except/finally no topo e logging por etapa.

5 · Fluxo automatizado

Tudo é disparado por python -m pim_report --periodo last (ou pelo cron, no dia da divulgação). O caminho feliz é uma linha reta; qualquer falha de rede desvia para o cache.

CLI / cron→ SidraClient (timeout+retry)→ cache (grava cru)→ schema (pandera)→ transform (4 leituras)→ analysis (narrativa)→ charts→ LaTeX → PDF+ HTML

Coleta → tratamento → análise → documento, sem intervenção manual. Em uma execução típica o processo leva poucos segundos — folgado para o prazo das 10h.

6 · Como o LaTeX é automatizado

  1. Template nota.tex.j2: um documento LaTeX com “buracos” preenchidos por Jinja2. Para não conflitar com as chaves {} do LaTeX, os delimitadores do Jinja são ((* ... *)) (blocos) e ((( ... ))) (variáveis).
  2. Render: render_latex.py injeta os dados/narrativa/gráficos no template e escreve o .tex preenchido. Valores em % passam pelos filtros |pct|tex (o % é comentário no LaTeX e precisa ser escapado).
  3. Compilação determinística: chama latexmk via subprocess; se ele não funcionar (ex.: MiKTeX sem Perl), faz fallback para pdflatex. Captura stdout/stderr e o .log.
  4. Falha tratada: se nenhum compilador gerar o PDF, preserva o .log e levanta LatexCompilationError com contexto — nunca um PDF parcial/silenciosamente errado.

Os gráficos são gerados pelo matplotlib em PDF vetorial (para o \includegraphics do LaTeX) e em PNG (embutido em base64 nos HTML).

7 · Robustez — minimizando o risco de “não funcionar no dia”

  • Timeout + retry: toda requisição tem timeout; tenacity re-tenta com backoff exponencial apenas erros transitórios (conexão/timeout e 5xx).
  • Validação de schema (pandera): se o IBGE mudar colunas/tipos, a esteira falha cedo e claro — em vez de produzir números errados.
  • Checagem de publicação: se o mês pedido ainda não saiu, retorna o Special Case PeriodoNaoPublicado e encerra com aviso — sem nota com mês velho.
  • Cache + degradação: cada execução grava o JSON cru; se a API cair, usa o cache mais recente e estampa um aviso destacado no PDF e nos HTML.
  • Build LaTeX tolerante: fallback de compilador e log preservado em falha.
  • Testes que forçam exceção: timeout, 5xx, JSON malformado, vazio e LaTeX quebrado têm testes dedicados (TDD — vermelho primeiro).
  • CI + cron + issue automática: a cada PR roda lint+testes; no dia, o cron executa a esteira e, em qualquer falha, abre uma issue automaticamente — o time descobre o problema antes do cliente.

8 · Bibliografia: Robert C. Martin — Código Limpo, Cap. 7 (Tratamento de Erro)

A arquitetura de erro segue o capítulo 7. Cada princípio é rastreável a um ponto do código:

Princípio 1Use exceções, não códigos de retorno

Toda falha levanta exceção; nenhuma função sinaliza erro com None/-1/flag.

Onde: sidra_client.py, transform.py, render_latex.py
raise SidraEmptyError('Consulta ao SIDRA sem registros de dados', ...)
Princípio 2Escreva try/except/finally primeiro (TDD)

Os testes que forçam exceção (Etapa 0) vieram antes da lógica feliz; o pipeline foi escrito ao redor de um try/except/finally.

Onde: tests/test_sidra_client.py, tests/test_render.py, pipeline.py
def test_timeout_e_traduzido_para_sidra_connection_error(...): with pytest.raises(SidraConnectionError): ...
Princípio 3Exceções não verificadas + hierarquia de domínio

Hierarquia própria sob uma raiz única PimReportError; o chamador captura amplo (PimReportError) ou cirúrgico (SidraHTTPError).

Onde: exceptions.py
class SidraError(PimReportError): ...
class SidraHTTPError(SidraError): ...
Princípio 4Forneça contexto nas exceções

A base carrega um dicionário 'contexto' e embute na mensagem; o wrapper usa raise ... from e (encadeamento) e o render preserva o .log.

Onde: exceptions.py, sidra_client.py, render_latex.py
raise SidraConnectionError('Falha de conexão/timeout ...', contexto={'url': url, 'timeout': self.timeout}) from erro
Princípio 5Defina classes segundo a necessidade do chamador

O cliente SIDRA é um WRAPPER que traduz requests.Timeout/HTTPError/JSONDecodeError/5xx/vazio em UMA família SidraError, isolando a dependência.

Onde: sidra_client.py, exceptions.py
except (requests.Timeout, requests.ConnectionError) as erro:
    raise SidraConnectionError(...) from erro
Princípio 6Defina o fluxo normal (Special Case pattern)

'Mês ainda não publicado' vira o objeto PeriodoNaoPublicado (publicado=False) em vez de if/else espalhado; o caminho feliz fica limpo.

Onde: special_cases.py, pipeline.py
@dataclass(frozen=True)
class PeriodoNaoPublicado:
    publicado = False
Princípio 7Não retorne None

Sucesso devolve coleção/objeto concreto; ausência de cache vira CacheVazioError; valores numéricos ausentes viram NaN (float), não None.

Onde: cache.py, transform.py, sidra_client.py
if not arquivos: raise CacheVazioError(...)
Princípio 8Não passe None

Parâmetros obrigatórios são validados na fronteira dos módulos com exigir_nao_nulo(), que levanta InvalidParameterError.

Onde: exceptions.py (guard) + todos os módulos
exigir_nao_nulo(tabela, 'tabela', operacao='buscar_tabela')

9 · Trechos de código comentados

Orquestração: try/except/finally no topo + degradação para cache (pipeline.py)

try:
    try:
        pacote, alvo = ... coletar ao vivo ...
        salvar_bruto(pacote, ...)            # grava o JSON cru de cada execução
    except SidraError as erro:               # API caiu?
        cache = carregar_mais_recente(...)   # usa o cache mais recente
        pacote, de_cache = cache.registros, True
    dados = construir_dados(pacote, ...)
    narrativa = gerar_narrativa(dados)
    pdf  = renderizar_nota(dados, narrativa, graficos, ...)
    html = renderizar_relatorio(dados, narrativa, graficos, ...)
except PimReportError as erro:
    logger.error('Esteira falhou: %s', erro, exc_info=True); raise
finally:
    logger.info('=== Fim da esteira PIM-PF ===')

Retentativa só em erros transitórios — backoff exponencial (sidra_client.py)

def _e_transitorio(exc):
    if isinstance(exc, SidraConnectionError): return True
    if isinstance(exc, SidraHTTPError):       return exc.transitorio  # 5xx
    return False

11 · Como rodar

# 1. ambiente
python -m venv .venv
.\.venv\Scripts\Activate.ps1        # Windows  (Linux/macOS: source .venv/bin/activate)
pip install -e ".[dev]"

# 2. gerar a nota do mês mais recente
python -m pim_report --periodo last

# 3. ou reprocessar um mês específico (a partir do cache)
python -m pim_report --periodo 202504

# testes e qualidade
pytest -q
ruff check .  &&  black --check .

Saídas em output/: nota_pim_AAAA-MM.pdf, o .tex preenchido, relatorio.html e este dossie.html. Neste exemplo: nota_pim_2026-04.pdf.
